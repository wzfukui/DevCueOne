# DevCue One 下一阶段产品与架构设计

## 文档状态

- 状态：锁定版
- 目标：作为下一阶段产品设计、架构拆分和实现排期的统一依据
- 适用范围：MVP 验证通过后的第一阶段演进
- 默认决策基线：以 [decisions.md](./decisions.md) 为准

## 输入来源

本设计基于以下输入合并而成：

1. 当前 MVP 已验证通过
   - 常驻监听
   - 语音转写
   - 调用 Codex CLI
   - 文本展示与语音播报拆分
   - 基础诊断和本地快捷动作
2. 本轮明确提出的未来需求
   - 展示所有历史会话
   - 激活和使用历史会话
   - 会话 A 任务进行中时，新建或切换会话 B 并启动新任务
   - 通过语音指令切换会话
   - 在新建会话时选择项目配置，并在会话内保持稳定项目上下文
   - 支持修改会话名称
   - 未来支持切换不同的 STT 和 TTS 接口
   - 未来支持用户选择工作语言
3. GitHub issue #1：拆分监听暂停与任务停止控制
   - 当前单一“停止监听”按钮语义混乱
   - 未来需要把“暂停语音输入”和“取消后台任务”彻底拆开

## 一句话定义

产品下一阶段不再是“单线程语音触发器”，而是一个：

`可管理多会话、可在创建时绑定稳定项目上下文、支持后台任务并行、支持本地会话控制的桌面 Codex 工作台。`

## 设计目标

### 核心目标

1. 用户能在同一个桌面应用中维护多条长期会话
2. 每条会话都能绑定清晰的项目上下文
3. 前台语音输入始终明确归属到当前激活会话
4. 后台 Codex 任务可以继续运行，不阻塞新会话
5. 控制行为的语义必须明确，不允许一个按钮承担多层含义
6. 所有关键状态都可追踪、可恢复、可诊断

### 非目标

1. 当前阶段不追求多设备同步
2. 当前阶段不引入复杂协作或云端会话共享
3. 当前阶段不支持一个会话同时拥有多个“主工作目录”
4. 当前阶段不把所有语音意图都交给 Codex 解释

## 产品原则

### 1. 当前激活会话是唯一语音输入目标

任何新语音输入都默认进入当前激活会话，不允许语音输入在多个会话之间自动漂移。

### 2. 任务和会话解耦

会话是用户视角的容器，任务是执行视角的实体。一个会话可以关联多次任务；任务状态不应该决定会话是否可见或可切换。

### 3. 项目上下文不是临时文本，而是显式对象

工作目录、开发工具、默认说明和使用备注，应该升格为“项目配置 profile”，而不是散落在输入框里的补充文本。

### 4. 语音能力应可替换

STT 和 TTS 不应永久写死为单一提供商。

产品应该允许未来切换：

- 不同 STT 提供商
- 不同 TTS 提供商
- 不同工作语言

### 5. 本地控制优先

会话切换、暂停监听、取消任务等动作，优先由本地路由识别并执行，不依赖 Codex 理解。

### 6. 默认行为必须安静且可预测

后台任务完成时默认不打断当前会话播报，只通过 UI 提示结果。语音输出只服务当前激活会话。

## 关键产品决策

## 1. 历史会话展示

会话列表将成为主导航，而不是附属日志。

每条会话至少展示：

- 名称
- 绑定项目
- 最近活动时间
- 最近一句摘要
- 当前是否有任务在跑
- 是否有未读结果

### 建议交互

- 支持按最近活动排序
- 支持固定重要会话
- 支持搜索会话标题和项目名
- 支持在列表内直接改名

## 2. 激活和继续使用历史会话

切回历史会话时，恢复的不只是聊天文本，还包括：

- 该会话的 `codex_thread_id`
- 绑定的项目配置
- 最近一次任务状态
- 最近转写结果
- 最近诊断事件

### 产品规则

- 默认行为是“继续当前会话”
- 可选提供“基于此会话新建分支会话”，但不属于本阶段必做

## 3. 多会话并行任务

### 用户目标

用户在会话 A 任务进行中，可以切到会话 B 继续说话并启动新任务。

### 产品规则

1. 当前激活会话接收输入
2. 后台会话任务可以继续执行
3. 后台任务完成后默认不自动播报
4. 后台完成通过 badge / toast / 状态点提示
5. 用户可在会话列表中看到哪些会话正在运行

### 架构含义

现有“全局单个 active task”模型必须升级为“按会话维护 task registry”。

## 4. 语音指令切换会话与本地控制

这类控制命令必须优先由本地路由识别，不经过 Codex。

### 示例意图

- 切到会话 `<名称>`
- 新建会话 `<名称>`
- 打开上一个会话
- 暂停语音输入
- 取消当前任务

### 路由原则

- 本地命中后直接执行
- 模糊命中时做确认
- 只有未命中的普通工作指令才进入 Codex

## 5. 项目配置与会话创建规则

### 本阶段明确决策

- 一个会话只绑定一个主项目配置
- 新建会话时必须明确选择一个项目配置
- 会话创建后不支持切换项目配置
- 已保存项目配置的名称和工作目录创建后固定
- 后续若要支持“参考目录”，作为扩展能力处理，不进入当前主流程

### 项目配置的价值

它不只是路径选择器，而是会话启动上下文的模板：

- 目录
- 显示名
- 可选开发工具
- 特定提示
- 使用备注

### 用户收益

- 减少每次重新交代项目背景
- 会话创建后上下文保持稳定
- 项目库和当前会话关系更清晰，不会在会话中途误改目录

### 当前交互形态

- 右侧“当前项目”卡片只展示当前会话绑定结果
- 项目库管理通过独立对话框完成
- 对话框左侧展示项目列表，右侧编辑项目配置
- 新建项目时优先选择工作目录，并自动带出名称
- 名称和工作目录只在新建草稿阶段可编辑

## 6. 会话名称可修改

### 明确决策

- 会话真实 ID 与显示标题分离
- 新会话默认自动命名
- 用户可随时修改标题
- 改名不影响任务归属和 thread 关联

## 7. 支持切换 STT / TTS 接口

### 本阶段设计要求

- STT 和 TTS 的选择权必须在模型层面预留
- UI 和持久化层都要允许未来配置：
  - STT provider
  - STT model
  - TTS provider
  - TTS model
  - TTS voice

### 产品收益

- 后续可接 OpenAI、系统本地能力、第三方服务或本地模型
- 测试时更容易切到 fake provider
- 不会把下一阶段架构锁死在单一服务商

## 8. 用户可选择工作语言

### 目标

允许用户显式设定自己当前的工作语言，而不是永远默认中文。

### 建议第一版范围

- 至少支持：
  - 中文
  - 英文
- 工作语言影响：
  - 默认转写语言
  - 默认 TTS 语言
  - 短确认语音包选择
  - UI 文案可选地后续跟进

### 设计要求

- 工作语言是显式设置项
- 会话级或全局级都应预留挂载点
- 切换语言后，新任务应使用新的默认语音策略

## GitHub Issue #1 的吸收方式

Issue #1 的核心不是按钮样式，而是“控制语义分层”。

这一点必须上升为产品规则：

### 控制层级

1. 语音输入层
   - 开始监听
   - 暂停监听
2. 会话层
   - 切换会话
   - 新建会话
   - 重命名会话
3. 任务层
   - 取消当前任务
   - 查看后台任务状态
4. 输出层
   - 停止当前播报

### 明确禁止

- 不允许一个按钮同时表达“暂停输入”和“取消任务”
- 不允许“停止监听”默认等于“杀掉一切后台任务”

### 对 UI 的要求

控制区必须让用户一眼知道：

- 现在暂停的是麦克风，还是
- 现在取消的是当前任务，还是
- 只是停止当前播报

这项要求直接纳入下一阶段设计，不再视为独立补丁。

## 推荐 UI 结构

## 左侧：会话列表

承载历史会话、状态和快速切换。

建议包括：

- 新建会话
- 会话搜索
- 会话列表项
- 会话状态点
- 绑定项目标签
- 直接重命名入口

## 中间：当前会话主区

承载：

- 当前会话消息
- 补充文本
- 语音输入主操作
- 当前会话任务状态

## 右侧：信息与诊断面板

承载：

- 当前会话元信息
- 当前项目摘要与项目管理入口
- 最近转写 / 最近音频
- 当前任务和后台任务摘要
- 诊断事件流

## 交互规则

- 语音输入入口只在当前会话主区可用
- 后台会话只显示状态，不抢焦点
- 播报只对当前会话生效

## 短确认语音策略补充

当前“收到”“我来看”这类短确认语音，不应长期依赖系统默认 TTS。

### 建议方向

1. 将短确认语音升级为“预生成本地音频资产池”
2. 资产可用高质量 API TTS 预生成，但运行时优先本地播放
3. 第一版至少支持：
   - 中文短确认
   - 英文短确认
4. 同一语义下保留多个变体，减少机械重复感
5. 若本地资产缺失，再回退到系统 / 浏览器 TTS

### 这样设计的原因

- 音质稳定，明显优于系统默认 TTS
- 不把每次确认语音都变成网络调用
- 响应更快，更适合作为“收到”级别的即时反馈
- 对自动化测试更友好，可直接验证命中哪条本地资产

### 边界

这项策略只针对“短确认语音”。

完整任务结果播报仍然遵循当前会话的正常播报策略。

## 持久化建议

下一阶段应尽快从“内存状态 + 零散文件”升级为结构化本地存储。

### 推荐方案

- 使用 SQLite 作为本地状态中心

### 理由

- 会话、任务、消息、项目配置和事件天然适合结构化存储
- 支持恢复、筛选、排序、统计
- 比 JSON 文件更容易维护一致性

### 推荐表

- `sessions`
- `tasks`
- `messages`
- `project_profiles`
- `event_logs`
- `settings`

### 文件系统保留项

- 原始音频文件
- 导出结果
- 临时转写中间件

但这些文件应在 SQLite 中保留索引关系。

## 架构建议

## 1. Main Process

负责：

- 任务生命周期
- Codex 子进程管理
- 音频文件持久化
- SQLite 读写
- 本地路由命令执行

## 2. Renderer

负责：

- 当前会话 UI
- 会话列表
- 项目配置管理
- 控制区展示
- 诊断可视化

## 3. Voice Router

独立模块，先于 Codex 运行。

负责：

- 会话切换命令识别
- 暂停 / 取消类命令识别
- 识别失败时回退到普通任务

## 4. Task Registry

独立维护每个会话下的运行任务。

至少需要：

- 按 `session_id` 查询运行中任务
- 允许多个会话同时有任务
- 精确取消当前会话任务
- 后台任务完成事件回传

## 日志与诊断策略

现有诊断区已证明价值，下一阶段应系统化。

### 目标

1. 用户能知道“哪条会话的哪次任务出了问题”
2. 开发时能快速重放一次失败路径
3. 日志不要再是全局混杂文本

### 建议

- 日志按 `session_id` 和 `task_id` 关联
- UI 上至少分为：
  - 当前会话诊断
  - 全局系统事件
- 对音频文件、转写结果、任务输出建立可点击关联

## 效率与性能方向

### 1. 后台任务静默完成

减少打断，提高多会话并行可用性。

### 2. 项目配置复用

减少重复输入路径、重复补充说明。

### 3. 本地命令优先

把低成本控制动作前置到本地，降低 Codex 负担和等待时间。

### 4. 恢复比重建更重要

历史会话恢复成功率，比一味优化初始界面更关键。

## 建议迭代顺序

## 第一阶段：数据与主导航成型

1. 本地持久化
2. 会话列表
3. 会话改名
4. 项目配置 profile

## 第二阶段：历史会话恢复

1. 恢复消息
2. 恢复 thread
3. 恢复会话对应项目

## 第三阶段：多会话后台任务

1. Task registry
2. 会话 A 跑任务、会话 B 继续输入
3. 后台完成提示
4. 取消动作精准化

## 第四阶段：本地语音控制

1. 切换会话
2. 暂停监听
3. 取消当前任务

## 第五阶段：诊断与体验细化

1. 事件流诊断
2. 更好的错误恢复
3. 更清晰的播报策略
4. UI 节奏与状态动效优化
5. 短确认语音资产化
6. 可切换 STT / TTS provider
7. 工作语言切换

## 已锁定的默认决策

以下决策已视为当前默认答案，不再作为开放问题悬而不决：

1. 多会话任务采用“会话级并行 + 全局并发上限 2”
2. 后台任务完成默认不播报，只做 UI 提示
3. 会话与项目配置采用“一会话一个主 profile”
4. 新建会话必须选择项目，创建后不支持切换
5. 已保存项目配置的名称和工作目录创建后固定
6. 控制区必须拆分监听 / 任务 / 播报三层语义
7. 本阶段持久化采用“本地 SQLite + 文件索引”

仍可保留为未来增强项，但不影响当前实现推进的问题包括：

1. 会话列表是否需要归档 / 收藏
2. 项目配置是否允许“推荐补充文本模板”
3. 会话是否需要“克隆为新会话”

## 结论

下一阶段的真正重点不是继续加零散按钮，而是先把产品的状态模型固定下来：

- 会话
- 任务
- 项目配置
- 本地控制路由
- 结构化持久化

GitHub issue #1 已被纳入统一设计，不再单独处理为局部 UI 修补。

只要以上模型先锁死，后续实现“历史会话、多任务并行、语音控制、项目管理、会话改名”会是顺滑演进；如果模型不先定，这些功能会互相打架。
