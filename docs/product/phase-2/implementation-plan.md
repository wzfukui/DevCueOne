# DevCue One Phase 2 实施计划

## 文档目的

这份文档把设计层结论翻译成实际可执行的实施顺序。

上游依据：

- [product-design.md](./product-design.md)
- [decisions.md](./decisions.md)

默认前提：

1. 多会话任务：会话级并行，全局并发上限 2
2. 后台任务完成：默认只做 UI 提示，不自动播报
3. 会话绑定：一会话一个主项目配置
4. 控制区语义：监听 / 任务 / 播报严格拆分
5. 持久化：本地 SQLite 为状态中心
6. 自动化测试默认采用 fake / stub / fixture，不走真实 OpenAI / Codex 主路径
7. 第一阶段测试平台至少覆盖 macOS

---

## 总体策略

实施顺序遵循两个原则：

1. 先做状态真相源，再做复杂交互
2. 先让会话和任务模型成立，再做语音命令和体验优化

换句话说：

- 先解决“存什么、怎么恢复”
- 再解决“怎么切、怎么并行”
- 最后做“怎么更顺手”

---

## Phase 2.0：基础设施重构

### 目标

把当前“单会话 + 单任务 + 内存状态”为主的结构，升级成可支撑多会话的底座。

### 需要完成的内容

1. 引入 SQLite
2. 定义基础表
   - `sessions`
   - `tasks`
   - `messages`
   - `project_profiles`
   - `speech_profiles`
   - `event_logs`
   - `settings`
3. 明确文件索引关系
   - debug audio path
   - transcript metadata
4. 建立主进程的数据访问层
5. 建立统一的 task registry

### 涉及模块

- [electron/main.mjs](../../../electron/main.mjs)
- 新增数据库访问模块
- 新增任务注册表模块
- IPC 协议扩展

### 完成标准

1. 应用重启后可恢复最近会话列表
2. 会话消息和任务状态不会因为 renderer reload 丢失
3. 每个任务都可关联到具体会话
4. 诊断事件可按 `session_id` / `task_id` 查询

### 风险

- 这是后续所有功能的基础，不能压缩成临时拼接
- 如果这里偷懒，后面会话切换和并发任务会反复返工

---

## Phase 2.1：会话列表与项目配置

### 目标

让“多会话”先在产品结构上成立。

### 需要完成的内容

1. 左侧会话列表
2. 会话创建
3. 会话自动命名
4. 会话改名
5. 会话激活
6. 项目配置 profile 列表
7. 会话绑定 / 改绑 profile
8. 当前会话元信息展示
9. 为未来挂接 speech profile 预留 UI 位置

### 建议 UI 结构

- 左侧
  - 新建会话
  - 搜索会话
  - 会话列表
- 中间
  - 当前会话消息区
  - 输入区
- 右侧
  - 会话信息
  - 绑定项目
  - 最近诊断摘要

### 完成标准

1. 用户能看到所有历史会话
2. 用户能直接改会话名称
3. 用户能为会话绑定一个项目配置
4. 切换会话时 UI 上下文同步更新

### 风险

- 会话标题和真实 ID 必须彻底分离
- profile 绑定逻辑必须在 UI 和持久化层保持一致

---

## Phase 2.2：历史会话恢复

### 目标

让会话切换不只是视觉切换，而是真正恢复到上次工作状态。

### 需要完成的内容

1. 恢复消息历史
2. 恢复绑定 profile
3. 恢复 `codex_thread_id`
4. 恢复最近任务状态
5. 恢复最近音频路径和转写结果
6. 恢复会话级诊断事件

### 完成标准

1. 切回旧会话时能看到完整消息
2. 当前输入会自动落到该会话
3. 若该会话已有 thread，可优先 resume
4. 用户能分辨“这个会话刚才发生了什么”

### 风险

- `session` 和 `thread` 的映射必须稳定
- 会话恢复不能只依赖 Codex 标准输出，要以本地数据库为准

---

## Phase 2.3：多会话后台任务

### 目标

满足“会话 A 跑着、会话 B 继续说”的核心需求。

### 需要完成的内容

1. 把当前全局 active task 改成会话级 task registry
2. 为任务增加状态
   - `queued`
   - `running`
   - `completed`
   - `failed`
   - `cancelled`
3. 支持不同会话同时启动任务
4. 加入全局并发上限
5. 超限任务进入队列
6. 会话列表展示运行状态和未读状态
7. 后台任务完成通过 UI 提示，不自动播报

### 完成标准

1. 会话 A 运行中时，会话 B 仍可发起任务
2. 两个任务的消息和诊断不串会话
3. 当前会话和后台会话的任务状态都可见
4. 后台任务完成后不会抢当前语音焦点

### 风险

- 取消逻辑必须精准定位到某个会话的某个任务
- 队列状态要在 UI 上明确区分，不能和“运行中”混为一谈

---

## Phase 2.4：控制区语义重构

### 目标

彻底解决 GitHub issue #1，建立可扩展控制模型。

### 需要完成的内容

1. 拆分监听控制
   - 开始监听
   - 暂停监听
2. 拆分任务控制
   - 取消当前会话任务
3. 拆分输出控制
   - 停止当前播报
4. 给每个动作提供独立状态反馈
5. 为未来增加更多控制按钮预留位置

### 完成标准

1. 用户可以明确知道停止的是哪一层
2. 暂停监听不会默认取消后台任务
3. 取消任务不会默认退出会话
4. 停止播报不会影响任务运行

### 风险

- 这是交互和状态机都要调整的点，不只是按钮换文案

---

## Phase 2.5：本地语音控制路由

### 目标

把会话切换、项目切换等指令前置到本地，不依赖 Codex 解释。

### 需要完成的内容

1. 建立 voice router
2. 支持本地命中以下指令：
   - 切到会话 `<名称>`
   - 新建会话 `<名称>`
   - 打开上一个会话
   - 切到项目 `<名称>`
   - 暂停监听
   - 取消当前任务
3. 模糊命中时二次确认
4. 未命中时才进入 Codex

### 完成标准

1. 基础控制命令可以脱离 Codex 完成
2. 错误命中率可接受
3. 命中后的 UI 状态变化明确

### 风险

- 本地命令路由必须可解释，不能做成黑盒
- 模糊匹配过 aggressive 会引入误切换

---

## Phase 2.6：诊断与恢复增强

### 目标

让产品在复杂场景下仍然可恢复、可解释、可维护。

### 需要完成的内容

1. 会话级诊断面板
2. 全局系统事件面板
3. 任务级错误归因
4. 音频、转写、任务输出之间的可追踪关联
5. 更清晰的失败恢复建议

### 完成标准

1. 用户能知道问题出在哪个会话、哪个任务
2. 开发者能快速回放失败链路
3. 问题不会再只表现为一段全局混杂日志

---

## Phase 2.7：确认语音体验优化

### 目标

把“收到”“我来看”这类短确认语音，从系统默认 TTS 升级为可控、低延迟、高质量的本地资产池。

### 需要完成的内容

1. 设计确认语音资产目录与索引结构
2. 准备中文短确认语音资产
3. 准备英文短确认语音资产
4. 支持同一语义下多个变体轮换
5. 运行时优先播放本地资产
6. 本地资产缺失时回退到系统 / 浏览器 TTS

### 完成标准

1. 收到语音后，短确认回复优先使用本地高质量音频
2. 不依赖每轮联网生成确认语音
3. 至少支持中英两类确认语音
4. 缺失资产时仍然可用，不阻塞主流程

### 风险

- 如果把确认语音做成“每轮实时 API TTS”，延迟和成本都会被放大
- 如果没有回退策略，资产缺失会直接伤害主链路可用性

---

## Phase 2.8：语音 provider 与工作语言切换

### 目标

让 STT / TTS 能力和工作语言成为可配置对象，而不是写死在当前实现中。

### 需要完成的内容

1. 定义 `speech_profiles` 数据模型
2. 支持切换 STT provider / model
3. 支持切换 TTS provider / model / voice
4. 支持设置工作语言
5. 把短确认语音包与工作语言联动
6. 为 fake provider 接入预留统一接口

### 完成标准

1. 用户可看到并修改当前语音配置
2. 新任务默认使用当前选择的 STT / TTS / 语言
3. 中文与英文工作语言至少可切换
4. fake provider 可被测试环境稳定接入

### 风险

- 如果 provider 切换只做成 UI 假选项，后续会反复返工
- 如果工作语言不入模型，只靠 prompt 猜测，行为会不稳定

## 推荐的数据层拆分

为了避免后续主进程继续膨胀，建议从一开始就拆模块。

### 1. `sessionStore`

负责：

- 创建会话
- 改名
- 激活会话
- 获取会话列表
- 更新最近活动时间

### 2. `taskStore`

负责：

- 创建任务
- 更新任务状态
- 查询会话运行状态
- 队列与并发上限

### 3. `messageStore`

负责：

- 记录消息
- 读取会话消息
- 生成列表摘要

### 4. `profileStore`

负责：

- 创建 / 编辑项目配置
- 查询 profile
- 会话绑定 / 改绑

### 4.1 `speechProfileStore`

负责：

- 创建 / 编辑 speech profile
- 查询当前默认语音配置
- 绑定 provider / model / language

### 5. `eventStore`

负责：

- 记录诊断事件
- 会话级筛选
- 任务级筛选

### 6. `voiceRouter`

负责：

- 本地命令匹配
- 模糊匹配确认
- 路由到会话 / 项目 / 控制动作

---

## 推荐的前端状态分层

Renderer 侧不要再把所有状态堆在一个 `App.tsx` 里。

建议至少拆成：

### 1. `workspace state`

- 当前激活会话
- 当前激活 profile
- 会话列表

### 2. `task state`

- 当前会话运行任务
- 后台会话运行任务
- 队列任务数量

### 3. `capture state`

- 是否监听
- 是否校准
- 是否正在播报
- 当前音量状态

### 4. `diagnostics state`

- 当前会话日志
- 全局事件
- 最近音频与转写

---

## 每阶段验收方式

建议每一阶段都用三层验收：

### 1. 功能验收

用户能不能完成这个阶段设计出来的动作。

### 2. 状态验收

重启、切换、后台任务完成后，状态是不是一致。

### 3. 诊断验收

失败后能不能看出问题属于哪条会话、哪个任务。

---

## 建议的里程碑划分

如果要真正推进，我建议把 Phase 2 切成 3 个里程碑。

### Milestone A：会话与数据底座

包含：

- SQLite
- session / message / profile / event 基础表
- tasks 最小 schema 与 store 骨架
- 会话列表
- 会话改名
- profile 绑定

结果：

- 多会话心智先成立

### Milestone B：恢复与并行

包含：

- thread 恢复
- 会话恢复
- task registry
- 多会话后台任务
- UI 状态点

结果：

- A 跑着、B 继续输成为现实

### Milestone C：控制与路由

包含：

- 控制区拆分
- 本地语音命令
- 诊断增强
- 播报策略细化
- 短确认语音资产化
- STT / TTS provider 切换
- 工作语言切换

结果：

- 产品从“能用”进入“顺手可控”

---

## 当前最推荐的下一步

如果你现在要继续往前推进，而不是停留在讨论层，我建议直接执行：

1. 先锁定 [decisions.md](./decisions.md) 里的 5 个默认答案
2. 以本文件中的 `Milestone A` 作为真正的开发起点

原因是：

- 没有数据底座，后面的多会话只是 UI 假象
- 先做语音切换会话，反而会把状态问题放大
- `Milestone A` 是收益最高、返工风险最低的一段
