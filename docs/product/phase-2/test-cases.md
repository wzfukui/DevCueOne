# DevCue One Phase 2 测试用例集

## 文档目的

这是一份**非代码测试用例集**。

目标不是实现自动化脚本，而是先锁定：

- 要测试什么
- 每项测试的前置条件是什么
- 预期系统行为是什么
- 预期可观察输出是什么

## 测试原则

### 1. 先保住 MVP，再扩展 Phase 2

Phase 2 任何改造都不能破坏当前 MVP 已验证通过的主链路：

1. 常驻监听
2. 语音转写
3. Codex 调用
4. 文本展示
5. 简短播报
6. 基础诊断

### 2. 先测用户看得见的结果，再测内部状态

每条用例至少要有两类结果：

- 用户可见结果：UI、提示、播报、列表、状态点
- 内部可观察结果：数据库记录、日志事件、任务状态、文件产物

### 3. 多会话特性必须重点防“串线”

Phase 2 最危险的问题不是“功能缺失”，而是：

- 消息串会话
- 任务串会话
- profile 串会话
- 诊断串会话

所以所有多会话测试都必须验证“归属正确”。

---

## 输出约定

每条测试执行后，建议统一记录：

- `结果`：通过 / 失败 / 阻塞
- `实际现象`
- `预期对比`
- `附件`
  - 截图
  - 诊断日志
  - 数据库快照
  - 调试音频文件路径

---

## 观测面

测试时建议至少观察以下 5 个面：

1. 主界面可见状态
2. 会话列表与会话主区
3. 右侧项目信息 / 任务信息 / 诊断
4. 主进程日志与 renderer 诊断日志
5. 本地持久化产物
   - SQLite
   - settings
   - debug-audio

## A. 当前 MVP 回归用例

这些用例必须在 Phase 2 开发开始前锁定，后续每个 milestone 完成后都至少回归一次。

### TC-R01 首次启动默认态

- 目标：
  - 验证首次启动时应用能正常进入可配置状态
- 前置条件：
  - 清空本地用户数据目录
- 操作：
  - 启动应用
- 预期行为：
  - 应用正常打开
  - 默认配置可见
  - 没有崩溃或白屏
- 预期输出：
  - 主界面可见
  - 默认 `workingDirectory` 已填
  - 默认系统提示消息存在

### TC-R02 保存设置

- 目标：
  - 验证设置可保存且重启后可恢复
- 前置条件：
  - 应用已启动
- 操作：
  - 修改 OpenAI Key、工作目录、启动后自动监听、全局静音等
  - 点击保存
  - 重启应用
- 预期行为：
  - 设置保存成功
  - 重启后值不丢失
- 预期输出：
  - UI 中显示保存后的值
  - 本地设置文件或数据库记录更新

### TC-R03 麦克风授权成功

- 目标：
  - 验证有权限时可进入监听
- 前置条件：
  - 系统层已允许麦克风
- 操作：
  - 点击开始监听
- 预期行为：
  - 成功进入监听态
- 预期输出：
  - 状态显示为监听中
  - 麦克风状态显示已授权

### TC-R04 麦克风授权拒绝

- 目标：
  - 验证拒绝权限时有清晰反馈
- 前置条件：
  - 系统层拒绝麦克风
- 操作：
  - 启动监听
- 预期行为：
  - 不进入可用监听态
  - 给出可理解提示
- 预期输出：
  - UI 显示权限失败原因
  - 不启动录音循环

### TC-R05 语音转写成功

- 目标：
  - 验证语音可转写并进入后续链路
- 前置条件：
  - OpenAI Key 有效
  - 麦克风可用
- 操作：
  - 说一条清晰中文指令
- 预期行为：
  - 语音转写成功
  - 生成用户消息
  - 进入 Codex 执行
- 预期输出：
  - 诊断区出现 `transcribe:start` 和 `transcribe:done`
  - 会话区出现对应用户消息

### TC-R06 空转写被忽略

- 目标：
  - 验证空白/无效转写不会污染消息流
- 前置条件：
  - 已进入监听
- 操作：
  - 触发极短无效音频
- 预期行为：
  - 不发起有效任务
- 预期输出：
  - 日志可见 `transcribe:empty ignored`
  - 不新增用户消息

### TC-R07 Codex 调用成功

- 目标：
  - 验证 Codex 返回成功结果时 UI 与播报正常
- 前置条件：
  - Codex CLI 正常可用
- 操作：
  - 发起一条可完成任务
- 预期行为：
  - assistant 消息追加
  - 简短 spokenReply 被播报
- 预期输出：
  - `turn:start`、`turn:done`
  - 会话区出现 assistant 消息
  - 当前会话收到播报

### TC-R08 手动取消任务

- 目标：
  - 验证取消动作能中断当前活动链路
- 前置条件：
  - 正在转写、正在 Codex、或正在播报
- 操作：
  - 点击当前取消入口
- 预期行为：
  - 当前活动目标被取消
  - UI 状态恢复
- 预期输出：
  - `cancelled: true`
  - 诊断中出现取消相关事件

### TC-R09 长任务默认不再被 90 秒硬中断

- 目标：
  - 验证默认配置下不再强制 90 秒超时
- 前置条件：
  - `CODEX_TURN_TIMEOUT_MS` 未设置
  - 构造一个超过 90 秒的任务
- 操作：
  - 启动任务并等待超过 90 秒
- 预期行为：
  - 任务持续运行，直到 Codex 自己结束或用户取消
- 预期输出：
  - 诊断日志中**不应**出现 `timeout: Codex 超过 90 秒未完成`

### TC-R10 本地快捷动作

- 目标：
  - 验证本地路由动作不必经过 Codex
- 前置条件：
  - 支持本地快捷动作的输入已定义
- 操作：
  - 触发“打开浏览器”“打开目录”“查看开放端口”等
- 预期行为：
  - 走本地动作
- 预期输出：
  - `backend=local`
  - 日志出现 `local:*`

### TC-R11 调试音频保留上限

- 目标：
  - 验证 debug-audio 不会无限增长
- 前置条件：
  - 连续完成超过 20 次录音
- 操作：
  - 检查调试音频目录
- 预期行为：
  - 只保留最近上限数量
- 预期输出：
  - 文件数不超过设定上限

## B. Milestone A 测试用例

目标：验证“会话 + profile + 持久化底座”。

### TC-A01 首次迁移创建默认会话与默认 profile

- 目标：
  - 验证旧版用户首次升级时可平滑进入新模型
- 前置条件：
  - 本地已有旧设置
  - 尚无 Phase 2 数据库
- 操作：
  - 启动新版本
- 预期行为：
  - 自动创建数据库
  - 自动生成默认 profile
  - 自动生成默认会话并绑定
- 预期输出：
  - `sessions` 至少 1 条
  - `project_profiles` 至少 1 条
  - 最近激活会话可见

### TC-A02 会话列表展示历史会话

- 目标：
  - 验证历史会话列表可见
- 前置条件：
  - 已存在多条会话
- 操作：
  - 打开应用
- 预期行为：
  - 左侧列表展示历史会话
- 预期输出：
  - 每条至少有标题、最近活动时间、摘要

### TC-A03 新建会话

- 目标：
  - 验证新建会话流程
- 前置条件：
  - 应用已打开
- 操作：
  - 点击新建会话
- 预期行为：
  - 新会话立即成为激活会话
  - 具备默认标题
- 预期输出：
  - 列表增加新项
  - 主区切换到新会话
  - 数据库新增 `session`

### TC-A04 会话重命名

- 目标：
  - 验证会话标题可直接修改
- 前置条件：
  - 至少一条会话存在
- 操作：
  - 进入重命名态
  - Enter 保存 / Esc 取消
- 预期行为：
  - 保存后标题更新
  - 取消时原值不变
- 预期输出：
  - UI 标题更新
  - `session.id` 不变
  - 重启后标题保留

### TC-A05 会话切换不串消息

- 目标：
  - 验证不同会话消息独立
- 前置条件：
  - 会话 A、B 均有消息
- 操作：
  - 在 A/B 间来回切换
- 预期行为：
  - 主区只显示当前会话消息
- 预期输出：
  - 不出现跨会话消息串入

### TC-A06 创建项目配置

- 目标：
  - 验证 profile 创建
- 前置条件：
  - 应用已打开
- 操作：
  - 打开项目管理对话框
  - 点击“新建项目”
  - 选择或输入工作目录
  - 确认名称自动带出后保存
- 预期行为：
  - 保存成功
- 预期输出：
  - profile 列表新增项
  - 数据库新增 `project_profiles` 记录

### TC-A07 编辑项目配置的固定字段与可编辑字段

- 目标：
  - 验证 profile 编辑边界
- 前置条件：
  - 已有 profile
- 操作：
  - 打开已有 profile
  - 检查名称与工作目录为只读
  - 修改项目开发工具、默认提示或使用备注并保存
- 预期行为：
  - 绑定该 profile 的会话显示同步更新
- 预期输出：
  - profile 数据更新
  - 名称与工作目录未被改写
  - 绑定关系未丢失

### TC-A08 新建会话时必须选择 profile

- 目标：
  - 验证会话创建入口强制选择项目
- 前置条件：
  - 已有至少一个 profile
- 操作：
  - 点击“新建会话”
  - 在未选择 profile 时观察创建按钮状态
  - 选择一个 profile 后创建会话
- 预期行为：
  - 未选择 profile 时不能创建
  - 创建成功后新会话自动成为当前激活会话
  - 后续输入默认使用所选 profile 的工作目录
- 预期输出：
  - 会话元信息显示所选 profile
  - `bound_profile_id` 正确写入

### TC-A09 已创建会话不可切换 profile

- 目标：
  - 验证项目只在创建时决定
- 前置条件：
  - 已存在绑定 profile 的会话
  - 项目库里存在多个 profile
- 操作：
  - 打开当前项目区域和项目管理对话框
  - 尝试寻找“切换项目”或“改绑项目”入口
- 预期行为：
  - 当前会话没有切换 profile 的入口
  - 浏览或编辑其他 profile 不会改写当前会话绑定关系
- 预期输出：
  - `bound_profile_id` 保持不变
  - 会话元信息中的当前项目不变化

### TC-A10 移除项目配置前需要二次确认

- 目标：
  - 验证危险操作保护
- 前置条件：
  - 已有一个被会话使用中的 profile
- 操作：
  - 点击“移除项目配置”
  - 先取消，再重新打开并确认移除
- 预期行为：
  - 第一次取消后数据不变
  - 确认后才真正删除
  - 若 profile 被会话使用，会给出解绑结果反馈
- 预期输出：
  - 取消后 profile 仍存在
  - 确认后 profile 消失
  - 相关会话的 `bound_profile_id` 被清空或相应更新

### TC-A11 重启后恢复最近激活会话

- 目标：
  - 验证工作台恢复能力
- 前置条件：
  - 已激活某条非默认会话
- 操作：
  - 关闭再启动应用
- 预期行为：
  - 自动回到最近会话
- 预期输出：
  - 当前激活会话正确
  - 标题 / 消息 / profile 同步恢复

### TC-A12 会话级元信息恢复

- 目标：
  - 验证会话摘要信息恢复
- 前置条件：
  - 会话已有 thread 摘要、最近活动时间
- 操作：
  - 重启后打开该会话
- 预期行为：
  - 右侧元信息完整
- 预期输出：
  - 标题、profile、最近活动时间、thread 摘要可见

## C. Milestone B 测试用例

目标：验证多会话后台任务与队列。

### TC-B01 会话 A 运行中，会话 B 继续启动任务

- 目标：
  - 验证会话级并行
- 前置条件：
  - 会话 A 已有长任务运行中
  - 当前全局运行数小于并发上限
- 操作：
  - 切到会话 B，发起新任务
- 预期行为：
  - B 任务成功启动
  - A 继续跑
- 预期输出：
  - `tasks` 中两条不同 `session_id` 的 `running`
  - 两边消息与日志不串线

### TC-B02 超过并发上限进入队列

- 目标：
  - 验证队列可见
- 前置条件：
  - 已有 2 个运行中任务
- 操作：
  - 再启动第 3 个任务
- 预期行为：
  - 第 3 个任务不立刻运行
  - 状态为 `queued`
- 预期输出：
  - UI 可区分 `queued` / `running`
  - 数据库状态正确

### TC-B03 运行任务完成后队列自动推进

- 目标：
  - 验证排队调度
- 前置条件：
  - 至少 1 个 queued 任务
- 操作：
  - 等待一个 running 任务结束
- 预期行为：
  - queued 任务自动转为 running
- 预期输出：
  - 任务状态迁移正确
  - 有对应事件日志

### TC-B04 后台任务完成默认不播报

- 目标：
  - 验证后台静默策略
- 前置条件：
  - 当前激活会话为 B
  - 后台 A 任务完成
- 操作：
  - 等待 A 完成
- 预期行为：
  - 不打断 B 的当前交互
- 预期输出：
  - 无自动语音播报
  - 会话 A 列表项有完成提示或未读标记

### TC-B05 多会话诊断不串线

- 目标：
  - 验证日志归属正确
- 前置条件：
  - A/B 并行跑任务
- 操作：
  - 查看两会话诊断
- 预期行为：
  - 每条诊断只归属自身会话 / 任务
- 预期输出：
  - `event_logs.session_id/task_id` 正确

## D. Milestone C 测试用例

目标：验证控制语义拆分、本地语音路由、诊断增强。

### TC-C01 暂停监听不取消任务

- 目标：
  - 响应 GitHub issue #1
- 前置条件：
  - 会话任务正在运行
  - 监听也处于开启状态
- 操作：
  - 点击“暂停监听”
- 预期行为：
  - 后续不再采集新语音
  - 当前任务继续执行
- 预期输出：
  - 监听状态变更
  - 当前任务保持 `running`

### TC-C02 取消任务不等于停止播报

- 目标：
  - 验证控制按钮职责清晰
- 前置条件：
  - 任务正在运行
- 操作：
  - 点击“取消当前任务”
- 预期行为：
  - 当前任务终止
  - 控制语义明确
- 预期输出：
  - 任务状态变 `cancelled`
  - 事件日志有取消记录

### TC-C03 停止播报不影响监听与任务

- 目标：
  - 验证语音输出控制独立
- 前置条件：
  - 当前会话正在播报
- 操作：
  - 点击“停止播报”
- 预期行为：
  - 播报立即停止
  - 监听与任务状态不被误改
- 预期输出：
  - speech 状态关闭
  - task / mic 状态维持原值

### TC-C04 语音切换会话

- 目标：
  - 验证本地路由命中
- 前置条件：
  - 至少两条会话存在
  - 其中一条名字可口头识别
- 操作：
  - 说“切到会话 XXX”
- 预期行为：
  - 本地识别并切换
  - 不把该句交给 Codex
- 预期输出：
  - 当前激活会话变化
  - 不产生新的 Codex 任务

### TC-C05 已废弃项目切换口令不应触发本地改绑

- 目标：
  - 验证旧项目切换口令不会破坏当前会话上下文
- 前置条件：

- 操作：
  - 说“切到项目 XXX”
- 预期行为：
  - 不触发本地项目切换
  - 当前会话绑定 profile 不发生变化
- 预期输出：
  - `bound_profile_id` 不变
  - 不出现本地项目改绑日志

### TC-C06 语音指令未命中时回退普通任务

- 目标：
  - 验证本地路由不会吞掉正常任务
- 前置条件：
  - 输入句式与控制命令相近但未命中
- 操作：
  - 说一句模糊命令
- 预期行为：
  - 回退给 Codex 作为普通任务
- 预期输出：
  - 生成正常用户消息和任务

### TC-C07 按会话查看诊断

- 目标：
  - 验证诊断视图按会话聚合
- 前置条件：
  - 多会话均有历史任务
- 操作：
  - 逐个切换查看诊断
- 预期行为：
  - 诊断视图跟随当前会话变化
- 预期输出：
  - 无跨会话事件污染

### TC-C08 诊断关联音频、转写、任务

- 目标：
  - 验证问题追溯能力
- 前置条件：

- 操作：
  - 打开某条任务诊断详情
- 预期行为：
  - 能看到音频、转写、任务状态之间的关联
- 预期输出：
  - 诊断中存在可关联的 task id / file path / transcript 事件

### TC-C09 短确认语音优先使用本地资产

- 目标：
  - 验证“收到”“我来看”等短确认语音不再默认依赖系统 TTS
- 前置条件：
  - 已准备本地短确认语音资产池
- 操作：
  - 触发一次正常语音输入
- 预期行为：
  - 系统优先播放本地确认语音资产
  - 不为确认语音单独发起实时网络请求
- 预期输出：
  - 可观测到命中的本地资产
  - 无确认语音专用网络调用

### TC-C10 短确认语音支持中英与回退

- 目标：
  - 验证中英短确认与回退策略
- 前置条件：
  - 中文和英文确认语音资产至少各一组
- 操作：
  - 分别触发中文和英文场景
  - 再人为移除一条资产，触发回退
- 预期行为：
  - 中文场景命中中文资产
  - 英文场景命中英文资产
  - 资产缺失时回退到系统 / 浏览器 TTS，不阻塞主流程
- 预期输出：
  - 资产命中记录正确
  - 回退路径可被日志或诊断识别

### TC-C11 切换 STT provider

- 目标：
  - 验证 STT provider 切换后，后续转写链路使用新的 provider
- 前置条件：
  - 系统已支持至少两个 STT provider，其中一个可为 fake provider
- 操作：
  - 切换 STT provider
  - 发起一次语音输入
- 预期行为：
  - 转写链路使用当前选中的 provider
- 预期输出：
  - 诊断或日志中可识别当前 STT provider

### TC-C12 切换 TTS provider

- 目标：
  - 验证 TTS provider 切换后，后续播报使用新的 provider
- 前置条件：
  - 系统已支持至少两个 TTS provider，其中一个可为 fake provider
- 操作：
  - 切换 TTS provider
  - 触发一次播报
- 预期行为：
  - 播报链路使用当前选中的 provider
- 预期输出：
  - 诊断或日志中可识别当前 TTS provider

### TC-C13 切换工作语言

- 目标：
  - 验证工作语言会影响默认语音行为
- 前置条件：
  - 至少支持中文和英文
- 操作：
  - 切换到英文
  - 再执行一次语音输入与确认语音流程
- 预期行为：
  - 默认转写语言、默认 TTS 语言、短确认语音包同步变化
- 预期输出：
  - 日志或 UI 可看出当前工作语言生效
