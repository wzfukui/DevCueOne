# DevCue One Phase 2 必须拍板的 5 个产品决策

## 文档目的

这份文档用于把 [product-design.md](./product-design.md) 收敛成真正需要拍板的少数关键决策。

原则是：

- 只保留会明显影响后续架构和交互的决策
- 每项都给出推荐默认值
- 每项都说明如果不先定，会带来什么代价

---

## 决策 1：多会话任务并发模型

### 要回答的问题

当会话 A 正在跑 Codex 时，会话 B 是否允许立即启动新的 Codex 任务？

### 选项

1. 完全串行
   - 全局任意时刻只能有一个 Codex 任务
2. 会话级并行
   - 不同会话可以并行执行任务
3. 会话级并行，但有全局并发上限
   - 不同会话可并行，但全局最多同时运行 N 个任务

### 推荐默认值

选项 3：`会话级并行，但设置全局并发上限`

推荐第一版：

- 当前激活会话可立即启动任务
- 其他会话也可启动任务
- 全局最多并发 `2` 个 Codex 任务
- 超出上限的新任务进入 `queued`

### 为什么这样选

- 满足你明确提出的 A 跑着、B 继续输的核心需求
- 避免机器上同时拉起过多 Codex 进程，导致 CPU / IO / 上下文竞争
- 架构上比“完全串行”更接近最终形态，但风险可控

### 如果不先定

- `Task Registry` 无法设计
- UI 不知道要显示“运行中”还是“排队中”
- 本地取消逻辑无法判断取消的是当前任务还是队列中的任务

## 决策 2：后台任务完成后的默认反馈方式

### 要回答的问题

当用户已经切到会话 B，而会话 A 的后台任务完成时，应用应该怎么反馈？

### 选项

1. 自动语音播报
2. 只在 UI 提示，不播报
3. UI 提示，且允许用户给会话开启“完成后播报”

### 推荐默认值

选项 2：`默认只做 UI 提示，不自动播报`

第一版建议行为：

- 当前激活会话才允许自动语音播报
- 后台任务完成后在会话列表展示状态点 / badge
- 可额外出现轻提示 toast，但不抢音频焦点

### 为什么这样选

- 多会话并行后，自动播报会很快变吵
- 用户正在会话 B 说话时，被会话 A 的播报打断，体验会立刻失控
- UI 提示足够表达“后台完成”，也更符合桌面工作台的安静风格

### 如果不先定

- 语音输出层和会话切换逻辑会互相冲突
- 后台任务完成策略会直接影响前端状态机设计

## 决策 3：会话和项目配置的绑定关系

### 要回答的问题

一个会话和项目目录之间，到底是什么关系？

### 选项

1. 一个会话可以随意切多个目录，不做显式绑定
2. 一个会话绑定一个主项目配置
3. 一个会话绑定多个项目配置，并支持同时作为上下文

### 推荐默认值

选项 2：`一个会话绑定一个主项目配置`

同时约束：

- 一个 `Session` 只绑定一个 `Project Profile`
- 一个 `Project Profile` 有一个主工作目录
- 用户可以改绑，但改绑属于显式动作
- 若用户想处理完全不同项目，产品应鼓励新建会话

### 为什么这样选

- 最符合用户心智：一个会话就是围绕一个项目持续推进
- 能显著降低上下文污染
- 数据模型简单，恢复也最稳定

### 扩展保留

后续可以加：

- 参考目录
- 只读上下文目录
- 关联资料目录

但不进入本阶段主流程。

### 如果不先定

- 会话恢复时无法可靠还原上下文
- “切项目”和“换会话”会混成一件事
- 语音切换项目的行为会很难定义

## 决策 4：控制区的语义拆分

### 要回答的问题

控制区是否明确拆开“暂停监听”“取消任务”“停止播报”？

### 选项

1. 继续维持单个复合按钮
2. 拆成多个独立按钮

### 推荐默认值

选项 2：`拆成多个独立按钮`

建议第一版至少拆出三种动作：

1. 监听控制
   - 开始监听
   - 暂停监听
2. 任务控制
   - 取消当前会话任务
3. 播报控制
   - 停止当前播报

### 为什么这样选

- 这正是 GitHub issue #1 的核心
- 多会话时代，一个按钮承载多个语义会直接造成误操作
- 后台任务并行后，“停止监听”绝不能等于“取消全部任务”

### 额外建议

控制区的状态描述也必须拆开显示，例如：

- `监听中`
- `当前会话任务运行中`
- `后台还有 1 个任务`
- `正在播报`

### 如果不先定

- 以后每加一个控制动作都要重写交互
- 用户很难建立稳定预期

## 决策 5：本阶段的数据持久化边界

### 要回答的问题

下一阶段的数据到底存到哪里，哪些算“真相源”？

### 选项

1. 继续以内存状态为主，零散落文件
2. 本地 SQLite 作为状态中心
3. 直接做远程同步 / 云端存储

### 推荐默认值

选项 2：`本地 SQLite 作为状态中心`

第一版边界建议：

- `SQLite`
  - sessions
  - tasks
  - messages
  - project_profiles
  - event_logs
  - settings
- 文件系统
  - 原始音频
  - 导出文件
  - 调试快照

并要求文件系统对象都能在 SQLite 里被索引到。

### 为什么这样选

- 历史会话、多任务、项目配置、事件流都天然适合结构化存储
- 恢复、搜索、排序、统计都需要数据库
- 远程同步现在会把问题复杂度放大太多，不适合这个阶段

### 如果不先定

- 会话历史无法可靠恢复
- 任务状态和消息展示容易出现不一致
- 日志无法按会话和任务追踪

## 推荐一次性拍板的默认答案

为了避免来回摇摆，建议直接把下面五项作为 Phase 2 默认答案：

1. 多会话任务采用 `会话级并行 + 全局并发上限 2`
2. 后台任务完成 `默认不播报，只做 UI 提示`
3. 会话与项目配置采用 `一会话一个主 profile`
4. 控制区 `必须拆分监听 / 任务 / 播报三层语义`
5. 本阶段持久化采用 `本地 SQLite + 文件索引`

## 如果你要我给出最务实的落地建议

我建议把这 5 个决策全部按上面的默认值直接锁死，不再反复讨论。

原因很简单：

- 它们已经足够支撑你要的 Phase 2
- 它们不会把实现复杂度抬到失控
- 它们都保留了未来继续演进的空间
