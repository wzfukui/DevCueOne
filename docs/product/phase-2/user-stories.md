## 用户画像

本阶段默认只有一个核心用户画像：

### 主用户

作为一个长期把 DevCue One 常驻在桌面上的开发者，我希望它像一个真正的工作台，而不是一次性的语音按钮。

这个画像决定了所有用户故事的优先级：

- 长期使用优先于一次性炫技
- 恢复能力优先于短期视觉新鲜感
- 可控性优先于自动化冒险

## Epic A：多会话导航

### US-A01 查看历史会话列表

- ID：`US-A01`
- Epic：多会话导航
- Milestone：A
- 用户故事：
  - 作为一个长期使用桌面代理的开发者，我希望在应用左侧看到所有历史会话列表，这样我能快速回到之前的工作上下文。
- 验收条件：
  - 左侧存在独立会话列表区域
  - 至少显示会话名称、最近活动时间、最近摘要
  - 当前激活会话有明确高亮
  - 应用重启后历史会话仍然存在
- 备注：
  - 这是 Milestone A 的主入口，没有它，多会话只是内部概念

### US-A02 新建会话

- ID：`US-A02`
- Epic：多会话导航
- Milestone：A
- 用户故事：
  - 作为一个开发者，我希望能快速新建一条会话，这样我可以把新的任务流和旧任务流分开。
- 验收条件：
  - 用户可从主界面直接触发“新建会话”
  - 新建会话前必须选择一个项目配置
  - 新会话创建后自动成为当前激活会话
  - 新会话具备默认标题
  - 新会话可被持久化保存
- 备注：
  - 新会话默认标题可后续被改名
  - 项目只在创建时决定，创建后不支持切换

### US-A03 直接重命名会话

- ID：`US-A03`
- Epic：多会话导航
- Milestone：A
- 用户故事：
  - 作为一个开发者，我希望直接修改会话名称，这样我能把会话整理成自己能识别的工作主题。
- 验收条件：
  - 用户可在会话列表中直接进入重命名状态
  - Enter 保存，Esc 取消
  - 改名后会话真实 ID 不变
  - 重启应用后名称仍保留
- 备注：
  - 标题来源需要区分自动命名和手动改名

### US-A04 切换会话

- ID：`US-A04`
- Epic：多会话导航
- Milestone：A
- 用户故事：
  - 作为一个开发者，我希望切换不同会话时主区内容立即切换，这样我能明确当前正在操作哪条工作流。
- 验收条件：
  - 点击会话列表项可以切换当前会话
  - 中间主区消息同步变化
  - 右侧会话元信息同步变化
  - 后续输入默认进入当前激活会话
- 备注：
  - 这是“当前激活会话是唯一输入目标”的产品规则落地

## Epic B：项目配置管理

### US-B01 创建项目配置

- ID：`US-B01`
- Epic：项目配置管理
- Milestone：A
- 用户故事：
  - 作为一个开发者，我希望创建项目配置，而不只是临时填写目录，这样我能长期复用路径、工具和项目说明。
- 验收条件：
  - 用户可以创建一个 profile
  - profile 至少包含名称和工作目录
  - 可选录入项目开发工具、默认补充说明、使用备注
  - 选择工作目录后可自动带出默认名称
  - profile 创建后会被持久化
- 备注：
  - profile 是目录的增强建模，不只是路径快捷方式
  - 项目库管理在独立对话框中完成

### US-B02 新建会话时选择项目配置

- ID：`US-B02`
- Epic：项目配置管理
- Milestone：A
- 用户故事：
  - 作为一个开发者，我希望在新建会话时就选定项目配置，这样后续输入天然落在正确的目录和项目背景里。
- 验收条件：
  - 每个会话创建时必须选择一个主 profile
  - UI 上能看到当前绑定的 profile 名称
  - 创建后后续输入默认使用该 profile 的工作目录
  - 切换会话时绑定 profile 同步恢复
  - 已创建会话不提供改绑入口
- 备注：
  - 本阶段坚持“一会话一个主 profile”

### US-B03 编辑项目配置

- ID：`US-B03`
- Epic：项目配置管理
- Milestone：A
- 用户故事：
  - 作为一个开发者，我希望编辑已有项目配置里的工具和元信息，这样我可以修正执行偏好和说明，而不需要反复新建。
- 验收条件：
  - 已保存 profile 的名称和工作目录为只读
  - 用户可编辑 profile 的项目开发工具、默认补充说明和使用备注
  - 编辑后已绑定该 profile 的会话会显示更新后的信息
  - 历史绑定关系不丢失
- 备注：
  - 这条 story 与 profile 复用强相关
  - 固定目录和名称是为了避免会话在使用中途漂移

### US-B04 移除项目配置前需要确认

- ID：`US-B04`
- Epic：项目配置管理
- Milestone：A
- 用户故事：
  - 作为一个开发者，我希望移除项目配置前先看到明确确认，这样我不会误删正在使用的项目。
- 验收条件：
  - 点击“移除项目配置”后先弹出确认对话框
  - 对话框会展示项目名称和受影响会话数
  - 用户确认后才真正移除
  - 若项目仍被会话使用，被解绑结果有清晰反馈
- 备注：
  - 这是危险操作保护，不属于普通表单保存

## Epic C：会话恢复

### US-C01 启动后恢复最近会话

- ID：`US-C01`
- Epic：会话恢复
- Milestone：A
- 用户故事：
  - 作为一个常驻用户，我希望应用启动后自动回到最近会话，这样我不需要每次重新定位到自己上次工作的地方。
- 验收条件：
  - 应用启动后自动恢复最近激活会话
  - 若无历史会话，则创建默认会话
  - 恢复后会话标题、消息、绑定 profile 可见
- 备注：
  - 这是“桌面工作台感”的核心体验之一

### US-C02 恢复会话消息历史

- ID：`US-C02`
- Epic：会话恢复
- Milestone：A
- 用户故事：
  - 作为一个开发者，我希望切回历史会话时看到这条会话之前的消息，这样我能接着做事，而不是重新开始。
- 验收条件：
  - 会话消息按 `session_id` 独立存储
  - 切换会话时只显示该会话自己的消息
  - 刷新或重启后消息不丢失
- 备注：
  - 这条 story 依赖持久化层成立

### US-C03 恢复会话级元信息

- ID：`US-C03`
- Epic：会话恢复
- Milestone：A
- 用户故事：
  - 作为一个开发者，我希望切回某条会话时看到它的项目、最近活动时间和 thread 摘要，这样我能快速判断上下文是否正确。
- 验收条件：
  - 当前会话元信息可展示
  - 至少包括标题、绑定 profile、最近活动时间、thread 摘要
  - 会话切换后元信息同步变化
- 备注：
  - 这条 story 有助于后续 thread resume 正常心智化

## Epic D：多会话后台任务

### US-D01 会话 A 运行时会话 B 可启动新任务

- ID：`US-D01`
- Epic：多会话后台任务
- Milestone：B
- 用户故事：
  - 作为一个开发者，我希望在会话 A 任务进行中切到会话 B 继续说话并启动新任务，这样我不用等一个任务结束才能继续工作。
- 验收条件：
  - 会话 A 有运行中任务时，会话 B 仍可启动任务
  - 两条任务分别归属自己的会话
  - 会话消息和诊断不会串线
- 备注：
  - 这是 Phase 2 的核心能力之一

### US-D02 超出并发上限时进入队列

- ID：`US-D02`
- Epic：多会话后台任务
- Milestone：B
- 用户故事：
  - 作为一个开发者，我希望当同时发起过多任务时，新任务进入可见队列，这样我能理解为什么它没有马上运行。
- 验收条件：
  - 超过全局并发上限的任务进入 `queued`
  - UI 上可区分 `queued` 和 `running`
  - 运行中任务完成后，排队任务自动推进
- 备注：
  - 默认并发上限由决策文档定义为 2

### US-D03 后台任务完成时不打断当前会话

- ID：`US-D03`
- Epic：多会话后台任务
- Milestone：B
- 用户故事：
  - 作为一个开发者，我希望后台会话任务完成时不要自动抢播报焦点，这样我在当前会话工作时不会被突然打断。
- 验收条件：
  - 后台任务完成后默认不自动语音播报
  - UI 上以 badge / 状态点 / toast 方式提示
  - 当前激活会话的播报策略不受后台完成事件影响
- 备注：
  - 这是多会话体验成立的关键约束

## Epic E：控制区语义拆分

### US-E01 暂停监听不等于取消任务

- ID：`US-E01`
- Epic：控制区语义拆分
- Milestone：C
- 用户故事：
  - 作为一个开发者，我希望“暂停监听”只影响麦克风输入，而不是顺带取消后台任务，这样我可以明确控制不同层级的行为。
- 验收条件：
  - 暂停监听后不再采集新语音
  - 已在后台运行的任务继续执行
  - UI 状态明确显示“监听已暂停”
- 备注：
  - 直接吸收 GitHub issue #1

### US-E02 单独取消当前会话任务

- ID：`US-E02`
- Epic：控制区语义拆分
- Milestone：C
- 用户故事：
  - 作为一个开发者，我希望单独取消当前会话任务，而不退出会话或关闭监听，这样我能精确停止错误执行。
- 验收条件：
  - 当前会话存在运行中任务时可单独取消
  - 取消动作不会误伤其他会话任务
  - 取消后会显示明确反馈
- 备注：
  - 需要依赖会话级 task registry

### US-E03 单独停止当前播报

- ID：`US-E03`
- Epic：控制区语义拆分
- Milestone：C
- 用户故事：
  - 作为一个开发者，我希望单独停止当前语音播报，而不影响任务执行，这样我能在不被打扰的情况下继续查看文本结果。
- 验收条件：
  - 停止播报只终止当前音频输出
  - 不影响任务状态
  - 不影响会话状态
- 备注：
  - 输出层必须和任务层分离

## Epic F：本地语音控制路由

### US-F01 通过语音切换会话

- ID：`US-F01`
- Epic：本地语音控制路由
- Milestone：C
- 用户故事：
  - 作为一个开发者，我希望直接说“切到会话 X”，应用就本地切换会话，这样我不需要手动点击或等待 Codex 理解控制指令。
- 验收条件：
  - 会话切换命令优先在本地命中
  - 命中后直接切换当前会话
  - 不把这条控制命令发送给 Codex
  - UI 上有明确切换反馈
- 备注：
  - 模糊命中时必须确认

### US-F02 项目切换口令不应改变当前会话项目

- ID：`US-F02`
- Epic：本地语音控制路由
- Milestone：C
- 用户故事：
  - 作为一个开发者，我希望当前会话的项目在创建后保持稳定，这样不会因为误说一句控制话术就把目录切错。
- 验收条件：
  - “切到项目 X”这类旧口令不再触发本地项目切换
  - 当前会话的绑定 profile 不会被隐藏逻辑改写
  - 若该输入继续进入普通任务链路，也不能改变会话绑定关系
- 备注：
  - 项目选择权只存在于新建会话阶段

### US-F03 未命中的控制指令回退到普通任务

- ID：`US-F03`
- Epic：本地语音控制路由
- Milestone：C
- 用户故事：
  - 作为一个开发者，我希望本地命令未命中时仍然按普通工作指令处理，这样不会因为本地路由存在就吞掉真实工作内容。
- 验收条件：
  - 本地路由未命中时，原始文本继续进入 Codex
  - 不产生无反馈失败
  - 命中和未命中的行为可通过诊断区区分
- 备注：
  - 这是 voice router 的容错底线

## Epic G：诊断与恢复

### US-G01 按会话查看诊断

- ID：`US-G01`
- Epic：诊断与恢复
- Milestone：C
- 用户故事：
  - 作为一个开发者，我希望查看当前会话的诊断事件，而不是一锅全局日志，这样我能知道问题到底属于哪条会话。
- 验收条件：
  - 诊断事件可按 `session_id` 展示
  - 当前会话和全局事件可区分
  - 每条事件可关联具体任务
- 备注：
  - 这会显著降低复杂场景下的排错成本

### US-G02 诊断里能关联音频、转写和任务

- ID：`US-G02`
- Epic：诊断与恢复
- Milestone：C
- 用户故事：
  - 作为一个开发者，我希望在诊断里追踪一条语音从原始音频到转写再到任务输出的链路，这样我能快速定位失败点。
- 验收条件：
  - 事件流中存在可关联的音频路径
  - 转写结果与任务启动事件可关联
  - 任务完成 / 失败可关联到同一条链路
- 备注：
  - 这是调试“语音代理”类产品的必要能力

## Epic H：确认语音体验

### US-H01 短确认语音优先使用本地高质量资产

- ID：`US-H01`
- Epic：确认语音体验
- Milestone：C
- 用户故事：
  - 作为一个长期使用桌面代理的开发者，我希望“收到”“我来看”这类短确认语音更自然，而且不依赖系统默认 TTS，这样整体体验会更稳定。
- 验收条件：
  - 收到语音后，短确认回复优先播放本地高质量音频资产
  - 不需要为每次确认语音单独发起网络请求
  - 若存在多个确认语音变体，可进行轮换
- 备注：
  - 该 story 只针对短确认语音，不等于完整任务结果播报

### US-H02 短确认语音支持中英与兜底回退

- ID：`US-H02`
- Epic：确认语音体验
- Milestone：C
- 用户故事：
  - 作为一个开发者，我希望短确认语音至少支持中文和英文，并在资产缺失时仍可用，这样不会因为音频资源问题阻塞主流程。
- 验收条件：
  - 至少存在中文短确认语音资产
  - 至少存在英文短确认语音资产
  - 本地资产缺失时可回退到系统 / 浏览器 TTS
- 备注：
  - 这是体验增强项，但不能破坏当前可用性

## Epic I：语音配置与工作语言

### US-I01 切换 STT provider

- ID：`US-I01`
- Epic：语音配置与工作语言
- Milestone：C
- 用户故事：
  - 作为一个开发者，我希望后续能切换不同的 STT provider，这样我可以根据成本、质量或本地环境选择合适的转写能力。
- 验收条件：
  - 系统存在可配置的 STT provider 入口
  - 新任务会使用当前选中的 STT provider
  - 测试环境可挂接 fake STT provider
- 备注：
  - 第一版不要求接很多 provider，但模型必须可扩展

### US-I02 切换 TTS provider

- ID：`US-I02`
- Epic：语音配置与工作语言
- Milestone：C
- 用户故事：
  - 作为一个开发者，我希望能切换不同的 TTS provider，这样我可以在系统默认、云端高质量语音和测试替身之间做选择。
- 验收条件：
  - 系统存在可配置的 TTS provider 入口
  - 当前会话播报使用选中的 provider
  - 测试环境可挂接 fake TTS provider
- 备注：
  - 该 story 与短确认语音资产化互补，但不是同一层能力

### US-I03 选择工作语言

- ID：`US-I03`
- Epic：语音配置与工作语言
- Milestone：C
- 用户故事：
  - 作为一个开发者，我希望选择当前工作语言，这样转写、播报和确认语音都能更匹配我的实际使用场景。
- 验收条件：
  - 用户可设置当前工作语言
  - 至少支持中文与英文
  - 工作语言会影响默认 STT 语言、默认 TTS 语言和短确认语音包
- 备注：
  - 这是未来国际化和多语工作流的基础
